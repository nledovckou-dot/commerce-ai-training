PRD
Product Requirements Document

Документ требований к продукту

Структурированный текст, который читает и человек, и AI-агент. Берётся сырьё — голосовые, переписка, ограничения, идея. На выходе — карта проекта с целями, юзерами, архитектурой, тестами и acceptance criteria. AI получает чёткую базу вместо «допиши что нужно».

01
Цели и не-цели
Что делаем в v1 и что сознательно не делаем.
02
Юзеры и сценарии
Персоны, JTBD, ключевые user stories.
03
Архитектура
Подсистемы, контракты, потоки, безопасность.
04
Acceptance
Тесты и критерий «v1 реально готов».

Этап PRD
Промпт 1 из 4

Генерация PRD из сырья

Ты — сильный product architect + tech lead + systems designer.
Твоя задача — превратить сырой материал в рабочий PRD, который
можно сразу отдать AI-агенту на разработку.

Контекст проекта:
- Название: [PRODUCT_NAME]
- Тип продукта: [PRODUCT_TYPE]
- Целевая аудитория: [TARGET_USERS]
- Цели: [GOALS]
- Ограничения: [CONSTRAINTS]
- Сырой материал: [RAW_INPUT]

Требования к работе:
1. Пиши по-русски.
2. Не делай маркетинговую воду. Нужен рабочий документ.
3. Если есть технологические или продуктовые факты,
   которые могут зависеть от текущего состояния инструментов,
   явно отделяй:
   - подтверждённые факты;
   - предположения;
   - open decisions.
4. Если какая-то секция не нужна, не удаляй её, а пометь
   "Не применимо" и коротко объясни почему.
5. Пиши так, чтобы этот PRD можно было сразу скормить
   AI-агенту / Codex без переписывания.
6. Будь конкретным: схемы, контракты, тесты, acceptance
   criteria, этапы разработки.
7. Если продукту нужны scoring systems, pipelines,
   jobs, background workers, rendering flows, auth,
   subscription, billing, moderation, privacy, AI layers
   или multi-step workflows — опиши это явно.
8. Для software products всегда давай не только
   product scope, но и execution-ready технический каркас.

Выведи PRD в следующей структуре:

# PRD — [PRODUCT_NAME]

## 1. Product summary
Коротко: что это за продукт, для кого, какой результат на выходе.

## 2. Problem statement
Какую проблему решаем и почему текущий процесс плох.

## 3. Goals
P0 / P1 / P2 цели.

## 4. Non-goals
Что в v1 сознательно не делаем.

## 5. Primary users / personas
Основной пользователь, вторичный пользователь, их контекст.

## 6. Core user stories / JTBD
Список ключевых сценариев.

## 7. Product principles
Какие правила должны определять продукт и реализацию.

## 8. Scope of V1
Что точно входит, что вне scope.

## 9. Assumptions and constraints
Технические, продуктовые, юридические, инфраструктурные
ограничения и рабочие допущения.

## 10. Proposed architecture
Высокоуровневая архитектура:
frontend / backend / workers / storage / AI providers /
local vs cloud / integrations.

## 11. Core workflows / pipelines
Пошаговые пайплайны системы от входа до результата.

## 12. Functional requirements
Разбей по подсистемам.
Для каждой подсистемы:
- что делает;
- входы;
- выходы;
- edge cases;
- user-visible behavior.

## 13. Internal decision systems
Если применимо:
scoring, ranking, routing, retries, quotas, guards,
fallback modes, approval logic.

## 14. Logical data model
Сущности, связи, ключевые поля, индексы, статусы.

## 15. Contracts
API / IPC / events / queue payloads / LLM JSON schemas /
render props / webhooks — что применимо проекту.
Приведи пример request / response / payload.

## 16. Security / privacy / permissions
Секреты, sandbox, PII, storage policy, role model,
internet access, audit trail.

## 17. Non-functional requirements
Performance, reliability, recovery, observability,
portability, offline mode, scalability — если актуально.

## 18. Acceptance criteria
Чёткий список, по которому можно сказать:
"v1 реально готов".

## 19. Test strategy
Unit / integration / e2e / smoke / golden / performance /
negative tests — что нужно именно этому продукту.

## 20. Suggested implementation breakdown
Epic → задачи → dependency-safe порядок внедрения.

## 21. Open decisions
Что ещё требует продуктового или технического решения.

## 22. First implementation slice
Какой минимальный кусок системы лучше реализовать первым,
чтобы быстро получить рабочий контур.

Стиль ответа:
- профессионально;
- конкретно;
- без воды;
- с явным разделением фактов, предположений и решений;
- так, чтобы документ был полезен и человеку, и AI-агенту.
~110 строк · 22 секции PRD
Что делает: превращает сырое описание (голосовые, переписка, идея) в структурированный PRD из 22 разделов. На входе — переменные [PRODUCT_NAME], [TARGET_USERS], [RAW_INPUT]. На выходе — документ, который можно сразу отдать AI-агенту без переспросов.

Этап Execution
Промпт 2 из 4

Основной execution prompt

From this message onward, work in execution mode.

Read and follow these files as your operating instructions:
- AGENTS.md
- docs/PLAN.md
- docs/STATUS.md
- docs/TEST_PLAN.md
- docs/BACKLOG.md

Goal:
Finish every unfinished task in docs/PLAN.md in dependency order.

Execution loop:
1. Select the first unfinished task whose dependencies are done.
2. Gather only the minimum context required for that task.
3. Implement that task with a scoped diff.
4. Run the validation commands defined for that task.
5. If validation fails, fix the issues and rerun validation
   until green.
6. Mark the task as done in docs/PLAN.md.
7. Append a concise log entry to docs/STATUS.md:
   - task finished
   - key files changed
   - commands run
   - result
   - next task
8. Immediately continue to the next task without waiting
   for my reply.

Operating rules:
- Do not stop after interim summaries.
- Do not ask me to continue.
- Do not spend turns on broad planning if the plan already exists.
- Keep diffs scoped to the current task.
- Do not do unrelated refactors.
- Do not claim completion without passing validations.
- Default to reasonable local assumptions and record them in
  docs/STATUS.md.
  Do not treat board sync as a blocker.
- If GitHub board / issue sync fails, log it and continue.
- Prefer working code, real tests, and real persistence over
  placeholders or TODO-shaped completion.
- If a task touches behavior, update or add tests accordingly.

Stop only if:
1. all tasks are done; or
2. a real blocker remains after 3 repair attempts; or
3. a secret / credential / manual action is required; or
4. an action is irreversible and needs explicit approval.

If blocked, output only:
1. blocker
2. what you tried
3. exact command or failing check
4. smallest action needed from me

Begin now with the first unfinished task.
~60 строк · execution loop
Что делает: переводит AI-агента в режим автономной работы по плану. Агент идёт milestone за milestone, валидирует каждый, обновляет STATUS.md и не останавливается на промежуточные «готово?». Останавливается только на реальном блокере, секретах или необратимом действии.

Этап Resume
Промпт 3 из 4

Continuation / resume

Resume execution mode.

Re-read and follow:
- AGENTS.md
- docs/PLAN.md
- docs/STATUS.md
- docs/TEST_PLAN.md
- docs/BACKLOG.md

Do not repeat finished work.
Do not restart from the beginning.
Use docs/PLAN.md and docs/STATUS.md to identify the exact next
unfinished task and continue from there.

Rules:
- If the previous stop was caused by a failed validation,
  start by fixing that failed validation, not by moving on.
- If there are partial changes for the current task, finish
  and validate them before switching tasks.
- Keep diffs scoped.
- Do not ask for permission to continue unless you hit a real
  blocker as defined in AGENTS.md.
- Continue task by task until all remaining tasks are done
  or a real blocker is hit.

If blocked, use the blocker format from AGENTS.md.
Otherwise, continue immediately.
~25 строк · возобновление
Что делает: возобновляет работу AI-агента после остановки или нового сеанса. Не переписывает план заново, не повторяет уже сделанные таски — читает STATUS.md и продолжает с точки остановки. Используется когда агент завис, чат закрыли, или нужно продолжить через час.

Этап Hardening
Промпт 4 из 4

Review · repair · hardening

Switch into review-and-repair mode.

Your task is to harden the current implementation against the PRD,
the plan, and the test plan before demo / release.

Read and follow:
- AGENTS.md
- docs/PLAN.md
- docs/STATUS.md
- docs/TEST_PLAN.md
- the PRD or spec file used by this project

Audit process:
1. Compare the current implementation against the PRD and plan.
2. Identify:
   - unfinished required behavior;
   - weak spots in reliability;
   - missing validations or tests;
   - UX or developer-flow regressions;
   - packaging / startup / migration / recovery risks.
3. Triage findings by release risk.
4. Fix the highest-risk gaps first.
5. Run the relevant validation suite.
6. Update docs/STATUS.md with:
   - what was audited;
   - what was fixed;
   - what remains deferred and why;
   - exact commands for final smoke-check.
7. Continue until the current build is demo-ready or a real
   blocker remains.

Rules:
- No speculative rewrites.
- No new features unless they are required by the PRD.
- Prefer reliability, correctness, and recoverability over
  cleverness.
- Do not stop after listing findings. Fix them.
- Keep changes reviewable and scoped.

Stop only when:
- the build is demo-ready according to docs/TEST_PLAN.md; or
- a real blocker remains after repair attempts.

If blocked, use the blocker format from AGENTS.md.
Otherwise, begin the audit and repair immediately.
~50 строк · перед демо
Что делает: переключает агента в режим «довести до релиза». Сверяет реализацию с PRD и тест-планом, находит слабые места, чинит самые рисковые сначала. Не добавляет новых фич — закрывает дыры. Запускается перед демо, лайвом или релизом.

GitHub

Что это даёт коммерческому отделу

Те же четыре механизма на языке менеджера: где живут материалы команды, как сохраняются изменения, как готовится новая версия и как ставятся задачи.

01 · Репозиторий
Единое место для всех материалов команды
Прайсы, скрипты звонков, шаблоны КП, инструкции для AI и регламенты лежат в одной структуре. Не нужно искать «актуальную» версию в чатах и почте — она всегда здесь.
папка sales-playbook с актуальным прайсом и шаблоном письма для дилера
02 · Коммиты
Каждое изменение подписано и обратимо
Видно: кто поменял, что именно поменял и зачем. К версии прайса от прошлой недели можно вернуться за два клика — без копий «прайс_финал_v3.xlsx».
«Иван, 14:32 — поднял минимальную скидку для региона ДВ на 2 п.п.»
03 · Ветки и Pull Request
Черновик правок с ревью руководителя
Готовите спецпредложение на ноябрь — работаете в отдельной «ветке», не трогая активный прайс. Руководитель видит все правки рядом со старой версией и одобряет их до публикации.
ветка promo-november ждёт апрува РОПа, потом уезжает в основной прайс
04 · Issues и Actions
Задачи команды и автоматические проверки
Issues — список задач со сроком и ответственным («обновить скидку на октябрь»). Actions — робот, который при каждом изменении прайса сам проверяет, что цены не упали ниже минимальной маржи.
issue #42 «пересчитать комиссию для топ-10 дилеров» — на Анне, до 5 ноября

Два способа подключить инструменты

Способ 1
API
Наш код вызывает чужой сервис по HTTP. Работает в продукте, не в чате.
